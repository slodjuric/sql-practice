# Adding a New Dataset

## Short version

1. Create `backend/src/data/datasets/<datasetKey>/raw/` and put your CSV files there.
2. Run `dataset:generate-config` to create `dataset.config.json`.
3. Review `dataset.config.json` and fix anything the generator got wrong.
4. Run `dataset:build-sample` to write import-ready files to `csv/`.
5. Run `dataset:import` to load the data into PostgreSQL.
6. Verify the tables.

> **You do not need to create or edit the `csv/` folder.**
> `csv/` is created and filled automatically by `dataset:build-sample`.
> Only put files in `raw/`.

All commands below run from inside `sql-practice/backend`.

---

## Folder roles

```
backend/src/data/datasets/<datasetKey>/
  raw/                  ← YOUR files go here. Never touched by scripts.
  csv/                  ← Generated by build-sample. Safe to delete.
  dataset.config.json   ← Generated by generate-config. Review before import.
```

| Folder / file | Who creates it | Who reads it | Safe to delete? |
|---|---|---|---|
| `raw/` | You | `generate-config`, `build-sample` | No — keep your source files here |
| `csv/` | `build-sample` | `import` | Yes — `build-sample` recreates it |
| `dataset.config.json` | `generate-config` | `build-sample`, `import` | Yes — `generate-config` recreates it |

`reset` deletes `csv/` and `dataset.config.json` but never touches `raw/`.

## Basic command flow

```bash
cd backend

npm run dataset:generate-config -- <datasetKey>   # reads raw/, writes dataset.config.json
npm run dataset:build-sample -- <datasetKey>       # reads raw/, writes csv/
npm run dataset:import -- <datasetKey>             # reads csv/, writes to PostgreSQL
```

## Reset and start again

If something went wrong or you want to start over for one dataset:

```bash
# See what will be deleted without deleting anything
npm run dataset:reset -- <datasetKey> --dry-run

# Delete generated files and DB schema for this dataset (asks for typed confirmation)
npm run dataset:reset -- <datasetKey>

# Same, without the confirmation prompt
npm run dataset:reset -- <datasetKey> --yes
```

Reset deletes:
- PostgreSQL schema and tables for `<datasetKey>`
- `csv/`
- `dataset.config.json`

Reset keeps:
- `raw/` and all your source CSV files
- All other datasets
- All app tables (`users`, `sessions`, `task_attempts`, `user_task_progress`, etc.)

After reset, run the three commands again from scratch.

## Available scripts

### `dataset:generate-config`

```bash
npm run dataset:generate-config -- <datasetKey>
npm run dataset:generate-config -- <datasetKey> --force   # overwrite existing config
```

Reads CSV files from `raw/`, infers column names, types, nullable flags, primary keys, foreign keys, load order, and `sampleOptions`, then writes `dataset.config.json`. **Always review the generated config** before running `build-sample` or `import` — auto-inference is best-effort (see [Common config fixes](#common-config-fixes)).

**What the generator infers automatically:**

- **Primary keys** — in priority order:
  1. A column named `id` that is unique and non-null (INTEGER)
  2. A column named `<singular_table_name>_id` that is unique and non-null (INTEGER) — e.g. `continent_id` for table `continents`, `country_id` for `countries`
  3. A column named `index` that is unique and non-null — fallback for bridge/fact tables
- **Foreign keys** — matches `<name>_id` columns to a table whose singular or plural form is `<name>`. References the parent's detected PK column. Skips columns that match the table's own entity ID pattern (prevents wrong self-references like `continents.continent_id → continents`).
- **Load order** — topological sort from FK dependencies.
- **sampleOptions** — `consistent` mode with the most-referenced table as `rootTable` and `limit: 100` when FK relationships exist; `preserve` mode for datasets with no FK relationships.

### `dataset:build-sample`

```bash
npm run dataset:build-sample -- <datasetKey>
npm run dataset:build-sample -- <datasetKey> --root <rootTable> --limit <N>
npm run dataset:build-sample -- <datasetKey> --mode preserve --limit <N>
npm run dataset:build-sample -- <datasetKey> --mode consistent --root <rootTable> --limit <N>
```

Reads CSVs from `raw/`, builds import-ready CSV files, and writes them to `csv/`. Behavior depends on `sampleOptions.mode` in `dataset.config.json` (or the `--mode` flag). See [Sample modes](#sample-modes).

### `dataset:import`

```bash
npm run dataset:import -- <datasetKey>
```

Creates the PostgreSQL schema and tables from `dataset.config.json`, then imports all CSV files from `csv/`. Existing tables for this dataset are dropped and recreated. See [Import options](#import-options) for how missing FK references are handled.

### `dataset:reset`

```bash
npm run dataset:reset -- <datasetKey>
npm run dataset:reset -- <datasetKey> --yes       # skip confirmation
npm run dataset:reset -- <datasetKey> --dry-run   # preview only
```

Drops the PostgreSQL schema, removes the datasets registry row, deletes `csv/` (recreates empty), and deletes `dataset.config.json`. Never touches `raw/`. `datasetKey` must match `/^[a-zA-Z0-9_-]+$/`; reserved schema names (`public`, `pg_catalog`, `information_schema`, etc.) are blocked.

## Common config fixes

After running `generate-config`, open `dataset.config.json` and check these manually:

**Primary keys not detected:** The generator auto-detects PKs for `id`, `<singular_table_name>_id`, and `index`. If none match, add `"primaryKey": true` manually and remove the `"nullable"` field from that column.

**Foreign keys not detected:** FK inference matches `<name>_id` to a table whose name is a plural or singular form of `<name>`. Most regular English plurals are handled (`country→countries`, `language→languages`, `person→people`). For unusual naming, add the FK manually:
```json
{ "name": "film_id", "type": "INTEGER", "nullable": false, "references": { "table": "films", "column": "id" } }
```
The `"column"` value should be the target table's PK column name as detected by `generate-config` (typically `id` or `<entity>_id`).

**NOT NULL columns with empty values:** If import fails with `Column "X" is NOT NULL but CSV value is empty`, change `"nullable": false` to `"nullable": true` for that column.

**INTEGER overflow:** PostgreSQL's `INTEGER` holds up to ~2.1 billion. The generator auto-upgrades a column to `BIGINT` if any value in the first 500 rows exceeds this. If the overflow only appears after row 500, you'll see an import error — change the column type from `"INTEGER"` to `"BIGINT"` manually, then re-run `import`.

**Wrong type inference:** The generator samples up to 500 rows. If a NUMERIC column is inferred as TEXT (e.g. due to corrupted values in the sample), fix the type manually.

## Sample modes

The `build-sample` script supports two modes, configured in `dataset.config.json`:

### Consistent mode

```json
"sampleOptions": {
  "mode": "consistent",
  "rootTable": "<rootTable>",
  "limit": 100
}
```

Selects `limit` rows from the root table, then expands bidirectionally along FK relationships until the set is stable:

- **Upward**: child rows pull in the parent rows they reference.
- **Downward**: selected parent rows pull in all their child rows.

Produces a relationally consistent sample with no broken FK references. Best for complete datasets. The `--root` and `--limit` CLI flags override config values.

### Preserve mode

```json
"sampleOptions": {
  "mode": "preserve",
  "limit": 100
}
```

Copies the first `limit` rows from each table's raw CSV directly into `csv/`, with no FK filtering. Missing FK references are left in the output and handled during import by `importOptions`. Use when you only have partial or independently-truncated CSV files. If `limit` is omitted, all rows are copied.

## Import options

Configure how the importer handles rows whose FK references are missing from the parent table:

```json
"importOptions": {
  "missingReferences": "nullify",
  "makeForeignKeysNullable": true
}
```

| Option | Values | Default | Effect |
|---|---|---|---|
| `missingReferences` | `"skip"` | `"skip"` | Rows with unresolvable FK references are dropped. |
| `missingReferences` | `"nullify"` | | Missing FK values are set to `NULL` instead of dropping the row. |
| `makeForeignKeysNullable` | `true` | `false` | FK columns are created as `NULLABLE` in PostgreSQL even if the config says `nullable: false`. Required when using `"nullify"` for NOT NULL columns. |

FK constraints are always kept in the schema. `INNER JOIN` on a nullified FK returns only matched rows; `LEFT JOIN` returns all rows.

## Workflow A — Complete dataset

Use when you have full source CSV files with consistent FK relationships.

```bash
mkdir -p src/data/datasets/<datasetKey>/raw
# copy your CSV files into raw/

npm run dataset:generate-config -- <datasetKey>
# Review dataset.config.json; see "Common config fixes" above

npm run dataset:build-sample -- <datasetKey>
npm run dataset:import -- <datasetKey>
```

## Workflow B — Partial dataset

Use when you only have partial CSVs where FK references may not align across tables.

After running `generate-config` and reviewing the config, add to `dataset.config.json`:

```json
"sampleOptions": {
  "mode": "preserve",
  "limit": 100
},
"importOptions": {
  "missingReferences": "nullify",
  "makeForeignKeysNullable": true
}
```

Then run `build-sample` and `import`. Rows with missing FK references will have that FK column set to `NULL` instead of being dropped.

## Troubleshooting

### `No such file or directory`

Running a path that includes `backend/` while already inside the `backend` folder. Confirm your location with `pwd`.

```bash
# Wrong (from inside backend/):
ls backend/src/data/datasets/<datasetKey>

# Correct:
ls src/data/datasets/<datasetKey>
```

### `raw/ directory not found`

The `raw/` folder was not created or source CSVs were not placed there.

```bash
mkdir -p src/data/datasets/<datasetKey>/raw
# Then copy your CSV files into that folder
```

### Config generated with `col_1`, `col_2`, ...

Source CSV files have no header row. Open `dataset.config.json` and rename the `col_N` names. Add `"sourceName"` so the importer knows which CSV column to read:

```json
{ "name": "film_id", "type": "INTEGER", "sourceName": "col_1" }
```

After renaming, re-run `build-sample` and `import`.

### Import fails: `Column "X" is NOT NULL but CSV value is empty`

Change `"nullable": false` to `"nullable": true` for that column in `dataset.config.json`, then re-run `import`.

### Import fails: `value "X" is out of range for type integer`

Change the column type from `"INTEGER"` to `"BIGINT"` in `dataset.config.json`, then re-run `import`.

### Import fails with FK constraint violation

Add to `dataset.config.json`:

```json
"importOptions": {
  "missingReferences": "nullify",
  "makeForeignKeysNullable": true
}
```

Then re-run `import`.

### Too few `INNER JOIN` results

Many FK columns are `NULL` because the sample is incomplete (Workflow B). Use `LEFT JOIN` to see all rows, or rebuild using Workflow A with complete source data.

### Numeric columns contain date-like values (e.g. `7.Jan`, `6.Apr`)

Excel auto-converted decimal numbers to date strings (e.g. `7.1` → `7.Jan`). Fix: repair the values in the raw CSV before importing — map month names back to numbers (`Jan=1`, `Feb=2`, ..., `Dec=12`), then re-run `build-sample` and `import`.

### CSV files use semicolons instead of commas

Pipeline parsers expect comma-delimited CSV. Convert before placing files in `raw/`:

```python
import csv, io

with open('input.csv', 'rb') as f:
    raw = f.read()
text = raw.replace(b'\r\n', b'\n').replace(b'\r', b'\n').decode('utf-8')
reader = csv.reader(io.StringIO(text), delimiter=';')
rows = [r for r in reader if any(f.strip() for f in r)]  # strip blank rows

with open('output.csv', 'w', newline='') as f:
    csv.writer(f).writerows(rows)
```
